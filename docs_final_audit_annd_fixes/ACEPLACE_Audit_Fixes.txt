I reviewed it as an audit + architecture fix.
Verdict
The repo has the right Phase 2 intent: it is envelope-driven in docs, uses Firestore as shared state between the Next.js tier and Python agent engine, has a public dispatcher in engine.ts, a parallel runner, a state machine, per-agent authority, and identity kernel files. That part is real and aligned with NXQ. 
The problem is that the implementation boundary is still wrong in a few critical places. The biggest one: engine.ts persists the envelope and then starts runEnvelopeParallel(...) inline and returns immediately, which is fine in a durable worker but fragile inside a web/API process. The same repo is a Next.js app, so that launch pattern is not a safe long-term runtime host. 
P0 fixes
1. Move runtime execution out of the Next.js request path
Current shape:
* Next.js is both UI/API and runtime dispatcher.

* engine.ts creates an envelope and starts the runner in-process. 

Fix:
   * Keep Next.js as control plane only.

   * Add a separate runtime-worker service whose only job is:

      * poll/claim runnable envelopes

      * acquire lease

      * execute steps

      * write traces/artifacts

         * Keep Python agent-engine as the LLM execution tier.

Target split:
apps/
  workstation-web/      # Next.js dashboard + API
  runtime-worker/       # long-lived TS worker
  agent-engine/         # FastAPI step executor


packages/
  runtime-core/         # envelopes, leases, state machine, #us#, persistence
  protocol/             # #us# types + validators
  acelogic-client/      # identity/license/continuity API client
That is the single most important fix.
2. Stop hardcoding the multi-agent identity map
Right now engine.ts seeds identity contexts from:
[agentId, "agent_researcher", "agent_grader"]
and does not include agent_worker. 
Fix:
            * Build identity_contexts from the actual planned steps or role assignments, not from a hardcoded array.

            * The planner should return the set of assigned agent IDs.

            * Envelope builder should derive contexts for every assigned agent before persistence.

Required rule:
identity_contexts = unique(step.assigned_agent_id for step in steps)
3. Remove the 
pending_verification
 execution bypass
The identity kernel currently allows execution to pass when the envelope fingerprint is "pending_verification" in at least two places. 
Fix:
               * Production rule: no execution without a canonical verified fingerprint.

               * If you need bootstrap mode, gate it behind an explicit environment flag:

ALLOW_PENDING_IDENTITY=false
and fail closed in prod.
Required rule:
if (!canonical_identity || fingerprint_mismatch) quarantine
if (expectedFingerprint === "pending_verification") reject in prod
4. Route quarantine through the state machine
per-agent-authority.ts directly updates envelope status to quarantined on fork detection. 
Fix:
                  * Never write terminal state directly from lease code.

                  * Lease layer should emit a domain error like FORK_DETECTED.

                  * Runtime layer should call the centralized state machine transition helper so every quarantine is traced consistently.

Your state machine already writes transition traces to execution_traces, which is the right model. 
5. Align auth docs and schema
API docs say runtime routes can authenticate using a Firebase ID token or a master API key stored in Firestore api_keys. The schema doc does not list api_keys at all. 
Fix:
                     * Add api_keys to the schema doc and code constants.

                     * Define exact fields:

interface ApiKeyDoc {
  key_id: string;
  org_id: string;
  label: string;
  hashed_key: string;
  status: "active" | "revoked";
  scopes: string[];
  created_at: string;
  last_used_at?: string;
}
Do not leave auth-critical storage implicit.
6. Lock down the Python agent engine
The FastAPI app currently uses allow_origins=["*"]. 
Fix:
                        * Restrict CORS to the workstation web origin and runtime worker origin.

                        * Require an internal service token between runtime-worker and agent-engine.

                        * Do not expose the agent engine as a broad public API.

async function claimNextEnvelope() {
  const snapshot = await db
    .collection("execution_queue")
    .where("status", "==", "queued")
    .limit(1)
    .get();


  if (snapshot.empty) return null;


  const doc = snapshot.docs[0];


  await doc.ref.update({
    status: "claimed",
    claimed_at: new Date().toISOString(),
  });


  return doc.data();
}
