Validate Runtime Invariants These invariants must hold under all test cases: one envelope is the single source of truth one running step may not share the same agent identity as another running step worker batch kill the runtime process while steps are in running Expected behavior: heartbeat stops leases expire dead-step recovery reclaims stale steps reclaimed steps reset to ready no duplicate artifacts are created envelope resumes and completes Success condition: crash does not permanently wedge the envelope Case B — Lease expires or is denied mid-step Test: simulate guard failure simulate lease renewal failure simulate lease instance mismatch Expected behavior: step fails or retries safely envelope state remains valid no partial corruption of step graph no silent continuation after invalid authority Success condition: authority failure produces safe halt or retry, not undefined execution Case C — High fan-out (50+ workers) Test: create large decomposition plan run 50+ worker steps in one envelope Expected behavior: no two concurrent steps use the same agent identity per-agent leases remain isolated Firestore transactions succeed without corrupting envelope grader waits until all required worker steps complete aggregation works correctly throughput scales without duplicate execution Success condition: system remains deterministic under parallel load 6. Determinism Validation Run the same envelope twice with: same input same role assignments same agents same planner behavior Expected output characteristics: same step graph shape same dependency structure same decomposition structure same artifact schema same trace lineage structure The exact wall-clock ordering may differ where parallelism is valid, but the execution structure must remain equivalent. If repeated runs diverge structurally, determinism is incomplete. 7. Telemetry Validation Confirm telemetry is sufficient to answer all of these: how many steps executed? how many steps failed? how many steps retried? how many dead steps were recovered? how many messages were stored? how many artifacts were created? what is per-agent execution count? what is per-agent failure count? what is lease renewal success vs failure rate? what is average step duration? (If duration is not yet recorded, add it to telemetry now.) If these cannot be answered, telemetry is incomplete. 8. Recovery Validation Validate the recovery system independently. Recovery must: scan envelopes in executing inspect running steps detect stale claims verify lease expiry or invalid lease reclaim step safely increment retry count mark ready if retry budget remains mark failed if retry budget exhausted transition envelope to failed if critical path can no longer progress Recovery must never: reclaim healthy work with an active lease duplicate an already completed artifact mutate dependencies incorrectly 9. Lease / Heartbeat Validation Validate the lease system as its own subsystem. Confirm: lease acquisition is atomic lease is written to authority_leases[agent_id] lease validation happens before step execution heartbeat renews lease before expiry heartbeat stops on completion or failure lease release occurs at end of step fork attempt quarantines safely This must hold per agent, not per envelope. 10.
